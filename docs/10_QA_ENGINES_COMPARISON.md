# 三大问答引擎对比说明

版本：v1.0.0 | 日期：2026-06-22

---

本文档以文段形式串联 KBQA、GraphRAG、qa_engine 三者的工作流程与关系，帮助理解整个问答系统的架构设计。

---

## 1. KBQA — 模板问答引擎

当用户输入一个问题时，KBQA 会调用 LLM 一次性完成两件事：识别意图和抽取实体。比如用户问"糖尿病有什么症状"，LLM 会输出意图是 `disease_symptom`、实体是 `{糖尿病, disease}`。但这里有个风险——如果 LLM 挂了或者返回了无法解析的内容，KBQA 直接返回 None，没有任何兜底，这条链路就断了。

如果 LLM 成功返回了意图和实体，接下来进入实体归一化。因为 LLM 抽取的实体名可能和图谱里的规范名称不一致（比如用户说"糖友病"，图谱里存的是"糖尿病"），归一化器会用三级匹配策略（精确→子串→模糊）把实体纠正为图谱中的规范名称。

归一化完成后，Cypher 生成器根据意图从 16 种模板中选出对应的查询语句，比如 `disease_symptom` 对应 `MATCH (Disease)-[has_symptom]->(Symptom)`，把实体名填入参数位置。然后去 Neo4j 执行查询，拿到结果后，答案格式化器用模板拼成自然语言——"糖尿病的症状包括：多饮；多尿；消瘦"。整条链路是：LLM 分析 → 归一化 → 模板 Cypher → 查询 → 模板拼装回答。每一步都确定性强，但也死板——LLM 挂了就全挂，查不到就返回空。

## 2. GraphRAG — 子图检索 + LLM 生成引擎

GraphRAG 的思路不同。它不关心用户问的是什么意图，只让 LLM 从问题中抽取实体。比如"糖尿病和高血压有什么关系"，它只抽 `{糖尿病, disease}` 和 `{高血压, disease}`，不做意图分类。同样的问题：LLM 挂了就返回 None，没有兜底。

实体抽取成功后，同样走归一化（复用 KBQA 的归一化器），然后用这些实体去 Neo4j 做多跳检索——不限定关系类型，把实体周围 1-2 跳的所有邻居节点和关系全部捞出来，形成一个子图。比如围绕"糖尿病"可能捞出症状、并发症、科室、食物、检查项等几十个节点和上百条边。

这个子图本身不是答案，需要 ContextBuilder 把它整理成结构化的自然语言文本（"【Disease】糖尿病\n  症状: 多饮/多尿\n  并发症: 高血压\n  ..."），然后把这个文本和用户原始问题一起发给 LLM，让 LLM 读上下文后生成回答。整条链路是：LLM 抽实体 → 归一化 → 多跳检索子图 → 整理成文本 → LLM 生成回答。比 KBQA 灵活，能回答复杂问题，但更慢，且依赖 LLM 生成质量。

## 3. qa_engine — 统一编排引擎

qa_engine 是架在 KBQA 和 GraphRAG 之上的编排层。它不发明新的分析逻辑，而是复用 KBQA 的 LLM 引擎来做第一步分析，同时补上了 KBQA 和 GraphRAG 都没有的兜底机制。

当用户提问后，qa_engine 先尝试 Level 1——调用 KBQA 的 LLM 引擎做全 LLM 分析（意图+实体）。如果 LLM 挂了或者没返回意图，自动降级到 Level 2——LLM 只抽实体，意图用关键词匹配（问题里有"症状"就归为 `disease_symptom`）。如果 LLM 彻底不可用，降到 Level 3——用 AC 自动机在词典里暴力匹配实体，意图纯靠关键词。三层兜底保证了只要词典在，系统就能跑。

分析完成后进入路由判断。规则很简单：没有实体就直接 LLM 兜底回答；实体数 ≥2 或者问题含"和/与/区别/关系"等复杂词或者问题超过 20 字，走 GraphRAG 路径；其他情况走 KBQA 模板路径。

更关键的是自动回退机制。如果 KBQA 模板路径执行了查询但 Neo4j 返回空结果，qa_engine 不会直接告诉用户"暂无相关信息"，而是自动把问题重新路由到 GraphRAG 路径再查一次。如果 GraphRAG 也查不到，最终由 LLM 用自己的知识兜底回答。整条链路是：三级降级分析 → 路由选择 → 模板路径（失败则自动切 GraphRAG）→ 还失败则 LLM 兜底。KBQA 和 GraphRAG 各管各的一条路，qa_engine 负责选路、兜底、回退，保证用户永远能得到一个回答。

---

## 4. 三者关系总结

| 维度 | KBQA | GraphRAG | qa_engine |
|------|------|----------|-----------|
| 角色 | 模板问答引擎 | 子图检索+LLM生成引擎 | 统一编排引擎 |
| 分析方式 | LLM 意图+实体（一次调用） | LLM 只抽实体 | 复用 KBQA 的 LLM + 三级降级兜底 |
| 兜底机制 | 无（LLM 挂了返回 None） | 无（LLM 挂了返回 None） | 三级：全LLM → LLM实体+关键词 → 词典NER+关键词 |
| 查询方式 | 模板 Cypher（写死的 16 种） | 动态多跳检索 | 路由决定走哪个 |
| 回答方式 | 模板拼装 | LLM 读子图生成 | 哪个路径返回就用哪个 |
| 擅长 | 单实体、单意图 | 多实体、复杂关系 | 自动选择最优路径 |
| 路由逻辑 | 无（独立模块） | 无（独立模块） | 实体数+关键词+问题长度判断 |
| 自动回退 | 无 | 无 | 模板无结果→自动切 GraphRAG→还不行→LLM兜底 |
| 能否独立使用 | 可以 | 可以 | 可以（推荐入口） |
| 依赖关系 | — | 复用 KBQA 的 EntityNormalizer | 复用 KBQA + GraphRAG 的全部组件 |

**一句话总结：KBQA 和 GraphRAG 是两个工具，qa_engine 是拿着这两个工具的调度员。**
