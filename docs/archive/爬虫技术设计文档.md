# 医药知识图谱数据爬虫 — 技术设计文档

> **⚠️ 已归档** — 本文档为历史参考。当前爬虫实现见 `data_spider/`。

## 1. 概述

本爬虫从医药网站采集疾病相关结构化数据，用于构建以疾病为中心的医药知识图谱。爬虫输出 JSONL 格式文件（`data/medical.json`），每行一条疾病记录，供下游 `knowledge_graph/` 模块导入 Neo4j 图数据库。

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| 自包含 | 无需 MongoDB 等外部存储依赖，直接输出最终格式文件 |
| 可恢复 | 支持断点续爬（`--resume`），中断后不丢失已爬数据 |
| 可控 | 支持指定页码范围、调节请求速率、测试模式 |
| 兼容 | 输出格式与原有 `data/medical.json` 完全一致 |
| 模块化 | 配置、解析、爬虫逻辑、分词、入口分离为独立文件 |

### 1.2 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| HTTP 客户端 | `requests` | Session 复用，自动管理连接池 |
| HTML 解析 | `lxml.etree` | XPath 解析，高性能 |
| 并发症分词 | LangChain + Ollama (`qwen3:8b`) | 本地大模型智能切分，三级降级策略 |
| 数据格式 | JSONL | 每行一个 JSON 对象，流式写入 |

## 3. 数据源分析

### 3.1 目标网站

- **域名**: 医药网站疾病库
- **编码**: GBK
- **页面总量**: 约 11,000 个疾病页（page ID: 1 ~ 11000）
- **有效页面**: 约 8,800+ 个（部分 ID 无对应疾病）

### 3.2 每种疾病对应 8 个子页面

| 序号 | 子页面 | URL 模式 | 采集内容 |
|------|--------|----------|----------|
| 1 | 概述 | `/il_sii/gaishu/{page}.htm` | 疾病名称、分类、简介、属性（医保/治愈率/费用等） |
| 2 | 病因 | `/il_sii/cause/{page}.htm` | 发病原因 |
| 3 | 预防 | `/il_sii/prevent/{page}.htm` | 预防措施 |
| 4 | 症状 | `/il_sii/symptom/{page}.htm` | 症状列表 |
| 5 | 检查 | `/il_sii/inspect/{page}.htm` | 诊断检查项（含子页面解析） |
| 6 | 治疗 | `/il_sii/treat/{page}.htm` | 治疗方式 |
| 7 | 食物 | `/il_sii/food/{page}.htm` | 宜吃/忌吃/推荐食谱 |
| 8 | 药品 | `/il_sii/drug/{page}.htm` | 推荐药品、药品明细 |

### 3.3 单次爬取请求量

每个有效疾病 = 8 个子页面请求 + N 个检查项子页面请求

预估总请求量：`8,800 × 8 + ~35,000 检查项 ≈ 105,000+` 次请求

## 4. 模块详细设计

### 4.1 config.py — 配置中心

**职责**: 集中管理路径常量、HTTP 头、字段映射和停用词。

| 配置项 | 说明 |
|--------|------|
| `OUTPUT_PATH` | 输出文件路径（`data/medical.json`） |
| `PROGRESS_PATH` | 断点进度文件路径（`data_spider/.spider_progress.json`） |
| `HEADERS` | HTTP 请求头（User-Agent） |
| `ATTR_MAP` | 中文属性名 → 英文字段名映射（11 项） |
| `STRIP_FIELDS` | 需去空格的字段集合 |
| `SPLIT_FIELDS` | 需按空格分割为列表的字段集合 |
| `STOP_CHARS` | 症状噪声过滤字符集（英文字母 + 数字 + 中文姓氏首字） |

### 4.2 parsers.py — 页面解析器

**职责**: 从 HTML 文本中提取结构化数据。所有 `parse_*` 方法接收 HTML 文本，返回解析结果。HTTP 请求由 Spider 负责，解析器只做内容提取。

#### 4.2.1 parse_basic_info — 概述页解析

**XPath 选择器**:

| 字段 | 主选择器 | 备选选择器 |
|------|----------|----------|
| 疾病名称 | `//title/text()` → 分割 "的简介" | - |
| 分类 | `//div[@class="wrap mt10 nav-bar"]/a/text()` | `//div[contains(@class,"nav-bar")]//a/text()` |
| 简介 | `//div[@class="jib-articl-con jib-lh-articl"]/p/text()` | `//div[contains(@class,"jib-articl-con")]//p/text()` |
| 属性 | `//div[@class="mt20 articl-know"]/p` | `//div[contains(@class,"articl-know")]/p` |

**属性映射表**（中文 → 英文字段名）:

| 中文属性 | 英文字段 | 值处理方式 |
|----------|----------|-----------|
| 医保疾病 | `yibao_status` | 去空格 |
| 患病比例 | `get_prob` | 去空格 |
| 易感人群 | `easy_get` | 去空格 |
| 传染方式 | `get_way` | 去空格 |
| 就诊科室 | `cure_department` | 按空格分割为列表 |
| 治疗方式 | `cure_way` | 按空格分割为列表 |
| 治疗周期 | `cure_lasttime` | 去空格 |
| 治愈率 | `cured_prob` | 去空格 |
| 常用药品 | `common_drug` | 按空格分割为列表 |
| 治疗费用 | `cost_money` | 原样保留 |
| 并发症 | `acompany` | LLM 智能分词（三级降级） |

#### 4.2.2 parse_common — 通用文本解析

用于病因（cause）和预防（prevent）页面。

**逻辑**: 提取所有 `<p>` 标签文本 → 过滤空行 → 用 `\n` 合并。

#### 4.2.3 parse_symptom — 症状解析

**主选择器**: `//a[@class="gre"]/text()`

**后处理**:
- 过滤首字为英文字母、数字或常见中文姓氏的噪声项
- 过滤长度 ≤ 1 的单字
- 去重

#### 4.2.4 parse_inspect — 检查项解析

**两级解析**:
1. `parse_inspect(html)` 从疾病检查页获取检查项 URL 列表：`//li[@class="check-item"]/a/@href`
2. `parse_inspect_name(html)` 从检查项详情页 `<title>` 提取检查项名称

> 注意：`parse_inspect` 只返回 href 列表，二次 HTTP 请求和缓存管理由 `spider.py` 的 `_resolve_inspect_names()` 负责。

#### 4.2.5 parse_food — 食物解析

**选择器**: `//div[@class="diet-img clearfix mt20"]`

返回三个列表（按 div 顺序）：
- `do_eat`: 宜吃食物（第 1 个 div）
- `not_eat`: 忌吃食物（第 2 个 div）
- `recommand_eat`: 推荐食谱（第 3 个 div）

#### 4.2.6 parse_drug — 药品解析

**选择器**: `//div[@class="fl drug-pic-rec mr30"]/p/a/text()`

返回两个列表：
- `drug_detail`: 原始药品名（含厂商），如 `"通药制药青霉素V钾片(青霉素V钾片)"`
- `recommand_drug`: 提取括号内药品通用名并去重

#### 4.2.7 transform — 数据转换

`@staticmethod` 方法，将各解析器的原始结果合并为最终 JSON 记录。接收 `splitter` 参数用于并发症分词。

### 4.3 spider.py — 爬虫主类

**职责**: HTTP 请求、爬取流程控制、断点续爬、检查项二次解析。

| 方法 | 说明 |
|------|------|
| `get_html(url)` | HTTP GET + GBK 解码 + 重试 |
| `_resolve_inspect_names(hrefs)` | 检查项链接 → 名称，带缓存 |
| `_load_progress()` | 加载断点进度 |
| `_save_progress(done_pages)` | 保存断点进度 |
| `crawl(resume)` | 爬取主循环 |
| `_crawl_one(page)` | 单个疾病页的 8 子页面爬取 + 转换 |

**设计决策**:
- 解析器只接收 HTML 文本，不做 HTTP 请求 → 职责分离
- 检查项缓存放在 Spider 层（跨页面共享）
- `transform` 的 `splitter` 参数由 Spider 传入 → 解析器无需了解分词实现

### 4.4 word_splitter.py — 并发症分词

**职责**: 将并发症原始文本切分为疾病名称列表。

**三级降级策略**:

```
Level 1: LLM 分词（LangChain + Ollama qwen3:8b）
    ↓ 失败
Level 2: 词典匹配（DictCutter，最大双向匹配算法）
    ↓ 无词典
Level 3: 简单分隔符切分（按常见标点分割）
```

| 组件 | 说明 |
|------|------|
| `LLMWordSplitter` | 主类，管理三级降级 |
| `DictCutter` | 词典匹配分词器，使用 `dict/disease.txt` |

**LLM 提示词**: 要求模型只输出逗号分隔的疾病名，添加 `/no_think` 抑制 qwen3 的思考模式输出。

### 4.5 main.py — CLI 入口

**职责**: 解析命令行参数，创建 `MedicalSpider` 实例并启动爬取。

## 5. 数据转换（transform）

`transform()` 将各解析器的原始结果合并为最终 JSON 记录。

### 5.1 输出 Schema

```json
{
  "name": "string          — 疾病名称",
  "desc": "string          — 疾病简介",
  "category": ["string"]   ,
  "prevent": "string       — 预防措施",
  "cause": "string         — 发病原因",
  "symptom": ["string"]    ,
  "yibao_status": "string  — 医保状态",
  "get_prob": "string      — 患病概率",
  "easy_get": "string      — 易感人群",
  "get_way": "string       — 传播方式",
  "acompany": ["string"]   ,
  "cure_department": ["string"],
  "cure_way": ["string"]   ,
  "cure_lasttime": "string — 治疗周期",
  "cured_prob": "string    — 治愈率",
  "cost_money": "string    — 治疗费用",
  "common_drug": ["string"],
  "check": ["string"]      ,
  "do_eat": ["string"]     ,
  "not_eat": ["string"]    ,
  "recommand_eat": ["string"],
  "recommand_drug": ["string"],
  "drug_detail": ["string"]
}
```

### 5.2 字段完整性

并非所有疾病都有完整字段。缺失字段不写入 JSON（而非写入空值），与原始 `medical.json` 行为一致。

## 6. 容错与恢复机制

### 6.1 HTTP 请求重试

```
失败 → 等待 1s → 重试
再失败 → 等待 2s → 重试
第 3 次仍失败 → 返回空字符串，跳过该子页面
```

- 404 响应直接返回空（页面不存在），不重试
- 使用 `requests.Session` 复用 TCP 连接

### 6.2 断点续爬

**进度文件**: `data_spider/.spider_progress.json`

```json
{
  "done_pages": [1, 2, 3, 4, 5, ...]
}
```

- 每 50 页自动保存一次
- `Ctrl+C` 中断时立即保存
- `--resume` 参数加载进度文件，跳过已完成页码

### 6.3 异常隔离

- 单个疾病爬取失败不影响其他疾病
- 单个子页面（如食物页）解析失败，其他子页面数据仍保留
- 所有异常记录日志，不中断主循环

---

## 7. 速率控制

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--delay` | 0.3s | 基础请求间隔 |
| 随机抖动 | 0 ~ 0.3s | 叠加在基础间隔上，模拟人工访问 |
| 重试退避 | 1s × attempt | 失败后递增等待 |

默认配置下，预计爬取全部数据耗时约 12-15 小时。

## 8. 使用方式

### 8.1 环境准备

```bash
pip3 install -r requirements.txt

# 如需 LLM 分词，还需启动 Ollama 服务：
# ollama serve
# ollama pull qwen3:8b
```

### 8.2 运行命令

```bash
cd data_spider

# 爬取全部（1-11000 页）
python3 main.py

# 指定范围测试
python3 main.py --start 1 --end 100

# 测试模式（只打印到终端，不写文件）
python3 main.py --start 1 --end 5 --test

# 中断后恢复
python3 main.py --resume

# 自定义输出路径和速率
python3 main.py --output /tmp/medical.json --delay 0.5
```

### 8.3 CLI 参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `--start` | int | 1 | 起始页码 |
| `--end` | int | 11000 | 结束页码 |
| `--output` | str | `data/medical.json` | 输出文件路径 |
| `--resume` | flag | false | 启用断点续爬 |
| `--delay` | float | 0.3 | 请求间隔（秒） |
| `--test` | flag | false | 测试模式，只打印不写文件 |
| `--ollama-model` | str | `qwen3:8b` | Ollama 模型名称 |
| `--ollama-url` | str | `http://localhost:11434` | Ollama 服务地址 |

## 10. 与原版对比

| 对比项 | 原版 | 新版 |
|--------|------|------|
| 外部依赖 | MongoDB + pymongo | requests + lxml + LangChain/Ollama（可选） |
| 数据流 | spider → MongoDB → build_data → MongoDB → 导出 | spider → 直接输出 JSONL |
| 代码结构 | 单文件（data_spider.py） | 5 个文件，职责分离 |
| 断点恢复 | 不支持 | 支持（`--resume`） |
| 重试机制 | 无 | 3 次重试 + 退避 |
| 限速 | 无 | 可配置延迟 + 随机抖动 |
| 类名 | `CrimeSpider`（错误） | `MedicalSpider` |
| 并发症分词 | 无 / 简单切分 | LLM + 词典匹配 + 降级（三级） |
| 检查项解析 | 需先爬原始 HTML 存 MongoDB，再二次解析 | 实时解析 + 缓存 |
| CLI 支持 | 无 | argparse 参数化 |
| 测试模式 | 无 | `--test` 打印不写文件 |
