# 医药知识图谱智能问答系统 — 技术设计文档

> **⚠️ 已归档** — 本文档为历史参考。当前问答系统见 `qa_engine/` 和 `docs/05_DOMAIN_MODEL.md`。

## 1. 概述

本模块基于 Neo4j 医药知识图谱，实现面向用户的自然语言问答。用户输入中文医疗问题，系统自动识别意图、提取实体、查询图谱并返回结构化回答。

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| LLM 驱动 | 意图识别和实体抽取由 LLM（Ollama qwen3:8b）完成，取代传统规则匹配 |
| 模块化 | 8 个文件职责分离，配置、分析、查询、格式化各自独立 |
| 可靠降级 | 三级降级策略，Ollama 不可用时仍能正常工作 |
| 安全查询 | 参数化 Cypher 模板，防止注入攻击 |
| 可扩展 | 新增意图/实体类型只需修改配置，无需改动核心逻辑 |

### 1.2 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| 意图识别 + 实体抽取 | LangChain + Ollama (`qwen3:8b`) | 单次结构化 LLM 调用 |
| 实体归一化 | `rapidfuzz` + dict/ 词典 | 精确/子串/模糊三级匹配 |
| 图数据库 | Neo4j + `py2neo` | 执行参数化 Cypher 查询 |
| 降级 NER | `pyahocorasick` (AC 自动机) | LLM 不可用时的备选方案 |

### 1.3 与原版对比

| 对比项 | 原版 | 新版 |
|--------|------|------|
| 意图识别 | 关键词匹配（硬编码规则） | LLM 语义理解（降级保留关键词规则） |
| 实体抽取 | AC 自动机字典扫描 | LLM 抽取 + 词典归一化（降级保留 AC 自动机） |
| 代码结构 | 4 个根目录文件 | `qa_system/` 模块，8 个文件职责分离 |
| Cypher 查询 | 字符串格式化 `.format()` | 参数化 `$name`，防注入 |
| 答案格式 | 固定模板 | 模板 + 可选 LLM 润色 |
| Neo4j 连接 | 硬编码 host/port | CLI 参数可配置 |
| 扩展性 | 新增意图需改多个文件 | 改 config.py 配置即可 |

## 2. 系统架构

```
用户输入
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  llm_engine.py — LLM 语义分析                            │
│  单次 LLM 调用 → {"intents", "entities", "has_negation"} │
│  失败 → 触发降级                                         │
└──────────────────────────┬──────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  entity_normalizer.py — 实体归一化                        │
│  LLM 实体名 → dict/ 词典精确/子串/模糊匹配 → 规范名称     │
└──────────────────────────┬──────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  cypher_generator.py — Cypher 查询生成                    │
│  意图 + 实体 → 参数化 Cypher 模板（18 种）                │
└──────────────────────────┬──────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  graph_query.py — Neo4j 执行器                            │
│  py2neo 执行 Cypher → 返回原始结果                        │
└──────────────────────────┬──────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  answer_formatter.py — 答案格式化                         │

└──────────────────────────┬──────────────────────────────┘
                           │
                           ▼
                       回答输出
```

**编排由 `chatbot.py` 统一调度，`main.py` 提供 CLI 入口。**

## 4. 模块详细设计

### 4.1 config.py — 配置中心

集中管理所有配置常量：

| 配置项 | 说明 |
|--------|------|
| `NEO4J_URI/USER/PASSWORD` | Neo4j 连接参数 |
| `OLLAMA_MODEL/BASE_URL` | Ollama 模型和地址 |
| `ENTITY_DICTS` | 实体类型 → 词典文件路径映射 |
| `INTENT_TYPES` | 18 种意图定义（含所需实体类型） |
| `LLM_SYSTEM_PROMPT` | LLM 语义分析的系统提示词 |
| `FUZZY_MATCH_THRESHOLD` | 模糊匹配阈值（默认 80） |

### 4.2 llm_engine.py — LLM 语义分析

**核心设计**：单次 LLM 调用同时完成意图识别和实体抽取。

**System Prompt 设计**：
- 列举全部 6 种实体类型 + 18 种意图类型
- 要求输出严格 JSON：`{"intents": [...], "entities": [...], "has_negation": bool}`
- 末尾 `/no_think` 抑制 qwen3 的思考模式输出

**响应解析**：
1. `re.sub(r"<think>[\s\S]*?</think>", "")` 清理残留思考标签
2. `re.search(r"\{[\s\S]*\}")` 提取 JSON（兼容 markdown 代码块）
3. 校验意图/实体合法性

**LLM 输出示例**：
```
问: "糖尿病不能吃什么"
→ {"intents": ["disease_not_food"], "entities": [{"name": "糖尿病", "type": "disease"}], "has_negation": true}

问: "头痛发烧可能是什么病"
→ {"intents": ["symptom_disease"], "entities": [{"name": "头痛", "type": "symptom"}, {"name": "发烧", "type": "symptom"}], "has_negation": false}
```

### 4.3 entity_normalizer.py — 实体归一化

LLM 抽取的实体名可能与 Neo4j 中的规范名称有差异，需要归一化。

**三级匹配策略**：

| 级别 | 方法 | 复杂度 | 示例 |
|------|------|--------|------|
| 1 | 精确匹配 | O(1) | "糖尿病" → 在 disease.txt 中直接命中 |
| 2 | 子串包含 | O(n) | "高血压" → 匹配 "高血压性心脏病" |
| 3 | 模糊匹配 | O(n) | "糖尿" → rapidfuzz 匹配 "糖尿病" (score=80+) |

**类型修正**：若 LLM 标注类型错误（如将 "高血压" 标为 symptom），归一化器会在所有词典中查找并修正为正确类型。

### 4.4 cypher_generator.py — Cypher 查询生成

**设计决策**：使用模板方式而非 LLM 生成 Cypher。

理由：
- 18 种意图对应固定的查询模式，无需动态生成
- 模板可测试、可审计
- 参数化 `$name` 消除注入风险
- LLM 生成 Cypher 不可靠，可能产生无效查询

**查询模板示例**：

```cypher
-- disease_symptom
MATCH (m:Disease)-[r:has_symptom]->(n:Symptom)
WHERE m.name = $name RETURN m.name, r.name, n.name

-- disease_acompany（双向查询）
MATCH (m:Disease)-[r:acompany_with]->(n:Disease)
WHERE m.name = $name RETURN m.name, r.name, n.name
-- +
MATCH (m:Disease)-[r:acompany_with]->(n:Disease)
WHERE n.name = $name RETURN m.name, r.name, n.name
```

### 4.5 graph_query.py — Neo4j 执行器

`py2neo.Graph` 的薄封装层。接收参数化 Cypher + params，执行查询返回结果列表。

### 4.6 answer_formatter.py — 答案格式化

**两种模式**：

| 模式 | 说明 | 延迟 |
|------|------|------|
| `template`（默认） | 18 种确定性模板，快速可靠 | ~0ms |
| `llm` | 模板先生成 → LLM 润色为自然语言 | ~1s |

**模板示例**：
```
disease_symptom → "{疾病}的症状包括：{症状1}；{症状2}；..."
disease_not_food → "{疾病}忌食的食物包括有：{食物1}；{食物2}；..."
drug_disease → "{药品}主治的疾病有{疾病1}；{疾病2}，可以试试"
```

### 4.7 chatbot.py — 编排器

统一调度所有组件，实现三级降级策略。

## 5. 降级策略

```
         ┌─────────────┐
         │ LLM 可用？   │
         └──────┬──────┘
           Yes  │  No
         ┌──────▼──────┐      ┌───────────────────┐
         │ LLM 分析     │      │ Level 3: 词典 NER  │
         │ 意图+实体    │      │ AC 自动机扫描      │
         └──────┬──────┘      │ + 关键词意图分类    │
                │              └───────────────────┘
         ┌──────▼──────┐
         │ 意图有效？   │
         └──────┬──────┘
           Yes  │  No
    ┌───────────▼─┐  ┌─────────────────┐
    │ Level 1     │  │ Level 2          │
    │ 全 LLM      │  │ LLM 实体         │
    │             │  │ + 关键词意图分类  │
    └─────────────┘  └─────────────────┘
```

| 级别 | 触发条件 | 意图识别 | 实体抽取 |
|------|---------|---------|---------|
| Level 1 | Ollama 可用，LLM 返回有效 JSON | LLM | LLM + 词典归一化 |
| Level 2 | LLM 有实体但无有效意图 | 关键词规则 | LLM + 词典归一化 |
| Level 3 | Ollama 不可用 | 关键词规则 | AC 自动机词典扫描 |

Level 3 组件（AC 自动机）**懒加载**，仅在首次 LLM 失败时初始化，避免 ~200ms 冷启动开销。

## 6. 使用方式

### 6.1 环境准备

```bash
pip3 install -r requirements.txt

# LLM 模式（推荐）需启动 Ollama：
ollama serve
ollama pull qwen3:8b

# 确保 Neo4j 运行且已导入知识图谱数据
```

### 6.2 运行命令

```bash
cd KBQA

# 交互式问答（默认）
python3 main.py

# 单次问答
python3 main.py --question "糖尿病有什么症状"

# LLM 润色回答模式
python3 main.py --answer-mode llm

# 远程 Neo4j
python3 main.py --uri bolt://192.168.1.10:7687 --user neo4j --password mypass

# 调试模式（显示意图、实体、Cypher 等中间信息）
python3 main.py --debug
```

### 6.3 CLI 参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `--uri` | str | `bolt://127.0.0.1:7687` | Neo4j URI |
| `--user` | str | `neo4j` | Neo4j 用户名 |
| `--password` | str | `neo4j` | Neo4j 密码 |
| `--ollama-model` | str | `qwen3:8b` | Ollama 模型名称 |
| `--ollama-url` | str | `http://localhost:11434` | Ollama 服务地址 |
| `--answer-mode` | str | `template` | 回答模式：template / llm |
| `--question` / `-q` | str | 无 | 单次问答模式 |
| `--debug` | flag | false | 显示调试信息 |

### 6.4 调试模式输出示例

```
用户: 糖尿病有什么症状

[DEBUG] Level 1: 全 LLM 模式
[DEBUG] 意图: ['disease_symptom'] | 实体: {'disease': ['糖尿病']}
[DEBUG] Cypher: MATCH (m:Disease)-[r:has_symptom]->(n:Symptom) WHERE m.name = $name ... | 参数: {'name': '糖尿病'}
[DEBUG] 查询结果: 15 条

助理: 糖尿病的症状包括：多饮；多尿；多食；消瘦；疲乏无力；...
```
